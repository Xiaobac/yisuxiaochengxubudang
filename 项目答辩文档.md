# 易宿酒店管理系统 - 答辩文档

## 一、项目概述

### 1.1 项目背景
易宿酒店管理系统是一个现代化的酒店预订和管理平台，为用户、商户和管理员提供完整的酒店业务解决方案。

### 1.2 系统定位
- **用户端**：在线浏览酒店、查看房型、预订房间
- **商户端**：管理酒店信息、房型库存、处理订单
- **管理员端**：审核酒店上架、管理平台内容

### 1.3 项目规模
- **前端代码**：约 15,000 行 TypeScript/React 代码
- **后端代码**：约 3,000 行 TypeScript 代码
- **数据库**：13 张数据表，27+ API 接口
- **页面数量**：20+ 页面

## 三、为什么选择这些技术？

### 3.1 为什么选择 Next.js？

**1. SEO 优化**
- 酒店搜索页面需要被搜索引擎抓取
- 服务端渲染（SSR）让搜索引擎能够直接获取页面内容
- 比传统的 React SPA 更利于 SEO

**2. 性能优势**
- 服务端渲染提升首屏加载速度
- 自动代码分割，按需加载
- 图片优化、字体优化等内置功能

**3. 全栈能力**
- API Routes 替代单独的 Express 后端
- 前后端一体化，简化部署
- 统一的开发体验

**4. 现代化架构**
- App Router 是最新的路由系统
- 支持 React Server Components
- 文件系统路由，开发更直观

### 3.2 为什么选择 TypeScript？

**1. 类型安全**
- 编译时发现错误，减少运行时 bug
- 避免了类型相关的问题（如我们遇到的角色判断 bug）

**2. 开发效率**
- IDE 智能提示和自动补全
- 重构更安全（重命名、查找引用）

**3. 可维护性**
- 类型定义即文档
- 团队协作更容易理解代码

### 3.3 为什么选择 Prisma ORM？

**1. 类型安全的数据库操作**
- 自动生成 TypeScript 类型
- 数据库查询也有类型检查

**2. 数据库迁移管理**
- 版本控制数据库结构
- 团队协作时同步数据库更改

**3. 简化复杂查询**
- 关联查询更简单（include）
- 自动处理外键关系

**示例**：
```typescript
// 一次查询获取酒店及其相关数据
const hotel = await prisma.hotel.findUnique({
  where: { id: hotelId },
  include: {
    merchant: true,    // 关联商户信息
    location: true,    // 关联地理位置
    roomTypes: true,   // 关联所有房型
  },
});
```

### 3.4 为什么选择 Ant Design？

**1. 企业级 UI 组件**
- 专为后台系统设计
- 组件丰富（Table、Form、Modal 等）

**2. 国际化支持**
- 内置完善的中文语言包
- 符合国内用户习惯

**3. 主题定制**
- 支持深色/浅色模式
- 可自定义主题颜色

**4. 响应式设计**
- 自动适配移动端
- 组件天然支持响应式

## 四、核心功能实现

### 4.1 用户认证系统

#### JWT 双 Token 机制
- **accessToken**：1 小时过期，用于 API 请求
- **refreshToken**：7 天过期，用于刷新 accessToken

**为什么用双 Token？**
- 安全性：accessToken 短期过期，即使泄露影响有限
- 用户体验：refreshToken 自动续期，用户无需频繁登录

**Token 刷新流程**：
```
1. 用户登录 → 获取两个 Token
2. 使用 accessToken 访问 API
3. accessToken 过期（1小时后）
4. 自动使用 refreshToken 刷新
5. 获取新的 accessToken
6. refreshToken 过期（7天后）→ 需重新登录
```

#### 密码安全
- 使用 **bcrypt** 加密密码（10 轮盐值）
- 即使数据库泄露，也无法反推原始密码
- 每个密码都有唯一的随机盐值

**文件位置**：
- 登录 API：`/app/api/auth/login/route.ts`
- 注册 API：`/app/api/auth/register/route.ts`
- Token 刷新：`/app/api/auth/refresh/route.ts`

### 4.2 权限控制系统（RBAC）

#### 角色定义
系统定义了 3 种角色：
1. **USER**（普通用户）：浏览酒店、预订房间
2. **MERCHANT**（商户）：管理酒店、处理订单
3. **ADMIN**（管理员）：审核酒店、平台管理

#### 权限矩阵
| 功能 | 用户 | 商户 | 管理员 |
|------|------|------|--------|
| 浏览酒店 | ✓ | ✓ | ✓ |
| 预订房间 | ✓ | ✗ | ✗ |
| 添加酒店 | ✗ | ✓ | ✗ |
| 管理订单 | ✗ | ✓ | ✗ |
| 审核酒店 | ✗ | ✗ | ✓ |

#### 权限控制实现
**前端**：路由守卫（`useAuth` Hook）
- 检查是否登录
- 检查角色是否匹配
- 不匹配则重定向

**后端**：API 验证
- 验证 JWT Token
- 检查 Token 中的角色信息
- 拒绝无权限的请求

**文件位置**：`/app/hooks/useAuth.ts`

### 4.3 酒店管理功能

#### 商户创建酒店
1. 商户填写酒店信息（名称、地址、星级、设施等）
2. 上传酒店图片
3. 提交后状态为 **pending**（待审核）
4. 等待管理员审核

#### 管理员审核流程
1. 管理员在审核页面看到所有 pending 状态的酒店
2. 查看酒店详细信息
3. 可以选择：
   - **审核通过**：状态变为 `published`，用户可见
   - **审核拒绝**：状态变为 `rejected`，记录拒绝原因

#### 酒店状态流转
```
pending (待审核)
   ├── → published (已发布) → 用户可见
   ├── → rejected (已拒绝) → 商户可见拒绝原因
   └── → offline (已下线) → 临时下架
```

**关键问题解决**：
- 商户查询时显示所有状态的酒店（包括 pending）
- 公开查询时只显示 published 状态的酒店
- 这样商户能看到自己待审核的酒店

**文件位置**：
- 酒店 API：`/app/api/hotels/route.tsx`
- 审核服务：`/app/services/review.ts`
- 审核页面：`/app/admin/review/page.tsx`

### 4.4 房型与库存管理

#### 房型管理
- 每个酒店可以有多个房型（如：豪华大床房、标准双床房）
- 每个房型有基础信息：
  - 名称、描述、价格、折扣
  - 设施（早餐、WiFi、浴缸等）
  - 总库存数量
  - 房型图片

#### 库存管理（房型日历）
商户可以设置每一天的：
- **当日价格**：可以覆盖基础价格（如节假日涨价）
- **可售数量**：可以覆盖总库存（如部分房间维修）
- **开关房态**：临时关闭售卖

#### 可用数量计算
```
可用数量 = (当日配额 || 总库存) - 已预订数量
```

**示例**：
- 房型总库存：10 间
- 2月1日：设置配额 8 间（2 间维修），已订 3 间 → 可用 5 间
- 2月2日：未设置配额，已订 4 间 → 可用 6 间（10-4）

**文件位置**：
- 房型 API：`/app/api/hotels/[id]/room-types/route.tsx`
- 库存 API：`/app/api/room-types/[id]/availability/route.tsx`
- 日历页面：`/app/merchant/calendar/page.tsx`

### 4.5 订单管理系统

#### 预订流程
1. 用户选择酒店和房型
2. 选择入住日期和退房日期
3. 填写客人信息（姓名、电话、身份证、特殊需求）
4. 系统计算总价 = 房价 × 折扣 × 天数 × 房间数
5. 创建订单，状态为 `pending`
6. 库存自动减少

#### 订单状态
- **pending**：预订中（待商户确认）
- **confirmed**：已确认
- **completed**：已完成
- **cancelled**：已取消

#### 商户订单管理
- 查看所有酒店的订单
- 查看客人信息和特殊需求
- 更新订单状态
- 查看订单统计

**文件位置**：
- 订单 API：`/app/api/hotels/[id]/bookings/route.tsx`
- 订单服务：`/app/services/booking.ts`
- 订单页面：`/app/merchant/bookings/page.tsx`

### 4.6 商户仪表盘

#### 统计数据
- **今日订单数**：今天创建的订单数量
- **本月营收**：本月所有非取消订单的总金额
- **酒店总数**：商户拥有的酒店数量
- **待处理订单**：状态为 pending 的订单数

#### 数据可视化
1. **营收趋势图**（折线图）
   - 最近 7 天的每日营收和订单数
   - 使用 recharts 库

2. **预订数据**（柱状图）
   - 不同状态订单的数量分布

3. **房型分布**（饼图）
   - 各房型的预订数量和营收占比

**实现方式**：
- 从真实订单数据计算统计值
- 使用 dayjs 进行日期计算和过滤
- 使用 recharts 渲染图表

**文件位置**：`/app/merchant/dashboard/page.tsx`

## 五、数据库设计

### 5.1 核心数据表

#### 1. users 表 - 用户表
存储所有用户（普通用户、商户、管理员）

**主要字段**：
- `id`：用户 ID（主键）
- `email`：邮箱（唯一，登录凭证）
- `password`：密码（bcrypt 加密）
- `role_id`：角色 ID（1=用户, 2=商户, 3=管理员）
- `phone`：手机号
- `name`：姓名

#### 2. hotels 表 - 酒店表
存储酒店信息

**主要字段**：
- `id`：酒店 ID
- `merchant_id`：商户 ID（外键）
- `name_zh`：中文名称
- `name_en`：英文名称
- `address`：地址
- `star_rating`：星级（1-5）
- `status`：状态（pending/published/rejected/offline）
- `facilities`：设施（JSON 格式）
- `images`：图片（JSON 数组）
- `rejection_reason`：拒绝原因

**status 状态说明**：
- `pending`：待审核（新创建的酒店默认状态）
- `published`：已发布（审核通过，用户可见）
- `rejected`：已拒绝（审核未通过）
- `offline`：已下线（临时下架）

#### 3. room_types 表 - 房型表
存储房型信息

**主要字段**：
- `id`：房型 ID
- `hotel_id`：酒店 ID（外键）
- `name`：房型名称
- `price`：基础价格
- `discount`：折扣（0.80 = 8折）
- `stock`：总库存
- `amenities`：设施（JSON）
- `images`：房型图片（JSON 数组）

#### 4. room_availability 表 - 房型可用性表
存储每一天的库存和价格

**主要字段**：
- `id`：记录 ID
- `room_type_id`：房型 ID（外键）
- `date`：日期
- `price`：当日价格（覆盖基础价格）
- `quota`：当日配额（覆盖总库存）
- `booked`：已预订数量
- `is_closed`：是否关闭售卖

**唯一索引**：`room_type_id + date`（一个房型的一天只能有一条记录）

#### 5. bookings 表 - 订单表
存储预订订单

**主要字段**：
- `id`：订单 ID
- `user_id`：用户 ID（外键）
- `hotel_id`：酒店 ID（外键）
- `room_type_id`：房型 ID（外键）
- `check_in_date`：入住日期
- `check_out_date`：退房日期
- `guest_count`：房间数量
- `total_price`：总价
- `status`：订单状态
- `guest_info`：客人信息（JSON）

#### 6. roles 表 - 角色表
存储角色定义

**初始数据**：
```
id=1, name='USER'     (普通用户)
id=2, name='MERCHANT' (商户)
id=3, name='ADMIN'    (管理员)
```

### 5.2 数据表关系

```
users (用户)
  ├── 1:N → hotels (商户拥有多个酒店)
  └── 1:N → bookings (用户有多个订单)

hotels (酒店)
  ├── N:1 → users (属于某个商户)
  ├── N:1 → locations (属于某个城市)
  ├── 1:N → room_types (有多个房型)
  └── 1:N → bookings (有多个订单)

room_types (房型)
  ├── N:1 → hotels (属于某个酒店)
  ├── 1:N → room_availability (每天的库存)
  └── 1:N → bookings (被多个订单预订)

bookings (订单)
  ├── N:1 → users (由某个用户创建)
  ├── N:1 → hotels (预订某个酒店)
  └── N:1 → room_types (预订某个房型)
```

### 5.3 为什么这样设计？

**1. 角色表独立**
- 方便扩展新角色（如：超级管理员、客服）
- 可以为角色添加权限配置

**2. 酒店状态字段**
- 实现审核工作流
- 商户可以看到审核进度
- 管理员可以管理平台内容质量

**3. 库存表独立**
- 支持动态定价（节假日涨价）
- 支持动态库存（部分房间维修）
- 提高查询效率（不用计算总库存）

**4. JSON 字段**
- `facilities`、`amenities`：灵活的键值对，便于扩展
- `images`：数组存储，支持多图
- `guest_info`：灵活存储客人信息

**文件位置**：`/prisma/schema.prisma`

## 六、API 设计

### 6.1 RESTful API 规范

遵循 REST 设计原则：

| HTTP 方法 | 路径 | 说明 |
|-----------|------|------|
| GET | `/api/hotels` | 获取酒店列表 |
| GET | `/api/hotels/:id` | 获取单个酒店 |
| POST | `/api/hotels` | 创建酒店 |
| PUT | `/api/hotels/:id` | 更新酒店 |
| DELETE | `/api/hotels/:id` | 删除酒店 |

### 6.2 主要 API 接口

#### 认证相关（3 个）
- `POST /api/auth/register` - 用户注册
- `POST /api/auth/login` - 用户登录
- `POST /api/auth/refresh` - 刷新 Token

#### 酒店相关（6 个）
- `GET /api/hotels` - 获取酒店列表（支持筛选）
- `GET /api/hotels/:id` - 获取单个酒店详情
- `POST /api/hotels` - 创建酒店
- `PUT /api/hotels/:id` - 更新酒店信息
- `DELETE /api/hotels/:id` - 删除酒店
- `GET /api/locations` - 获取城市列表

#### 房型相关（4 个）
- `GET /api/hotels/:id/room-types` - 获取酒店房型列表
- `POST /api/hotels/:id/room-types` - 添加房型
- `GET /api/room-types/:id/availability` - 查询房型可用性
- `PUT /api/room-types/:id/availability` - 更新可用性

#### 订单相关（3 个）
- `GET /api/hotels/:id/bookings` - 获取酒店订单
- `POST /api/hotels/:id/bookings` - 创建订单
- `PATCH /api/bookings/:id` - 更新订单状态

### 6.3 API 响应格式

#### 成功响应
```json
{
  "success": true,
  "data": { ... }
}
```

#### 错误响应
```json
{
  "success": false,
  "error": "错误信息"
}
```

#### HTTP 状态码
- `200 OK`：成功
- `201 Created`：创建成功
- `400 Bad Request`：请求参数错误
- `401 Unauthorized`：未登录或 Token 过期
- `403 Forbidden`：无权限
- `404 Not Found`：资源不存在
- `500 Internal Server Error`：服务器错误

### 6.4 数据格式转换

**问题**：前端和后端使用不同的命名风格
- 前端：`snake_case`（hotel_id, check_in_date）
- 后端：`camelCase`（hotelId, checkInDate）

**解决方案**：在 Service 层统一转换

```typescript
// app/services/booking.ts
export const getMyBookings = async () => {
  const response = await get('/hotels/.../bookings');

  // 转换：camelCase → snake_case
  return response.data.map((booking: any) => ({
    hotel_id: booking.hotelId,
    hotel_name: booking.hotel?.nameZh,
    check_in_date: booking.checkInDate,
    check_out_date: booking.checkOutDate,
    total_price: Number(booking.totalPrice),
    // ...
  }));
};
```

**好处**：
- 前端组件不需要关心后端数据格式
- 修改格式只需改 Service 层
- 保持前后端各自的代码风格

### 7.3 遇到的核心问题

#### 问题 1：登录后跳转失败 ⭐
**现象**：
- 用户登录成功
- 停留在首页，显示"欢迎来到 Next.js"
- 没有跳转到对应角色的后台

**原因**：
```typescript
// 错误代码（/app/hooks/useAuth.ts）
if (currentUser.role !== requiredRole) {
  router.push('/');
}
```
- `currentUser.role` 是对象：`{id: 2, name: 'MERCHANT'}`
- `requiredRole` 是字符串：`'merchant'`
- 对象 !== 字符串，永远为 true

**解决**：
```typescript
const userRole = currentUser.role?.name?.toUpperCase();
const required = requiredRole?.toUpperCase();

if (required && userRole !== required) {
  router.push('/');
}
```

**教训**：
- 检查数据类型很重要
- TypeScript 类型定义能提前发现此类问题

#### 问题 2：商户看不到待审核酒店 ⭐
**现象**：
- 商户创建酒店成功（HTTP 201）
- 但在酒店列表中看不到
- 新酒店状态是 `pending`

**原因**：
```typescript
// /app/api/hotels/route.tsx
// 默认只返回 published 状态的酒店
if (!status) {
  where.status = 'published';
}
```

**解决**：
```typescript
// 商户查询：显示所有状态
// 公开查询：只显示已发布
if (status) {
  where.status = status;
} else if (!merchantId) {
  where.status = 'published';
}
```

**教训**：
- 不同角色的数据访问权限要区分
- 默认过滤要考虑各种场景

#### 问题 3：管理员无法获取待审核列表
**现象**：
- 调用 API 返回 404
- API 路径是 `/api/review/pending`

**原因**：
- 后端没有这个路由
- 应该用 `/api/hotels?status=pending`

**解决**：
```typescript
// /app/services/review.ts
export const getPendingHotels = async () => {
  // 修改 API 路径
  const response = await get('/hotels?status=pending');
  return response.data;
};
```

**教训**：
- 前后端 API 约定要统一
- 遵循 RESTful 风格更清晰

## 十、答辩要点

### 如果被问到技术选型

**问：为什么选择 Next.js 而不是 React？**

答：
1. **SEO 需求**：酒店搜索页面需要被搜索引擎抓取，SSR 比 CSR 更有利
2. **性能优化**：服务端渲染提升首屏加载速度

4. **开发体验**：文件系统路由、自动代码分割等内置功能

**问：为什么用 Prisma 而不是直接写 SQL？**

答：
1. **类型安全**：自动生成 TypeScript 类型，编译时发现错误
2. **迁移管理**：版本控制数据库结构，团队协作更方便
3. **简化查询**：关联查询更简单，自动处理外键关系
4. **防 SQL 注入**：使用参数化查询，自动防护
